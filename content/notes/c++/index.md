# namespace
 namespace是命名空间,可以自己打包一些函数,类,变量等放进去,防止名字冲突,如:

```c++
namespace MyLib{
    int swap(){
        
    }
}
```

此后在使用时可以通过`std::swap`来调用。平时使用的`using namespace std`其实是直接全局 using 声明(将整个命名空间引入),而std就是 *C++* 标准委员会为整个标准库指定的命名空间名字,包括`vector`,`cin`等等。namespace最大的作用就是防止名字冲突,所以一般不建议直接全局 using 声明

# 三目运算符
`(condition ? expr1 : expr2)`该结构可以做到简写简单的`if`判断,`condition`第一个bool如果为真则执行**表达式**`expr1`,否则执行**表达式**`expr2`。注意要求两个分支的类型能互相转换,比如不可以`expr1`是`char`,`expr2`是`int`。同时cpp中可以直接使用1代表`true`,0代表`false`。

# 指针,引用

下面介绍指针和引用:

指针pointer(以`*p`为例子,他是`a`的指针)是一个存储的值是另一个对象或函数在内存中的地址的变量,就像一个指向房子(内存位置)的地图,指引编译器找到位置。

其中`p`就是内存地址(如`0x5ffe3c`),而`*p`可以直接当成对象操作。比如`*p = 0;`可以直接使得pp变为0,指针可以嵌套,也就可以有指向指针的指针。
- 赋值方式是:`int *p = &a;`,其中`&`表示取地址
- 可以为空值(`nullptr`)
- 可以改变指向
- 拥有简写语法糖例如`(*p).replace(0, 5, "hi");`可以写作`p->replace(0, 5, "hi");`

    抽象的,`p->member`就是`(*p).member`的简写,可以将通过成员访问运算符简单完成成员访问和操作

- int* 指针加1,地址增加 sizeof(int) 个字节,因为指针算术以指向的类型大小为单位。这样设计是为了方便数组遍历。同理对于char*就是sizeof(char) = 1个字节。
- 在大多数表达中,数组名会“退化”为指向其首元素的指针,比如:
    - `
    int arr[5] = {1, 2, 3, 4, 5};
    int *p = arr;   // arr 退化为 &arr[0]
    `
    除非使用`&arr`将会得到整个数组的指针(类型为 int(*)[5]),不是 int**
    - 数组下标本质就是指针运,编译器将 arr[i] 理解为 *(arr + i),也就是`arr[2] == *(arr + 2)` 完全等价。**这就意味着,甚至 2[arr]也能编译通过,因为 2[arr] → *(2 + arr) → *(arr + 2),不过这种写法纯属语法恶搞**
    - 当数组作为函数参数时,实际上传递的是指针,所以在函数内部,sizeof(arr) 得到的是指针大小,不是数组大小。通常需要额外传入长度参数。


引用reference(以`&q`为例子,他是`qq`的引用)是一个已存在变量的别名

其中`qq`就是引用的

- 赋值方式是:`int &pp = p;`
- 不可为空
- 不可改变(改为引用别的变量)

两者最大的区别在于`&`,这是一些`&`的用法:

- 表达式中的 &：取地址符(对指针和引用都适用)

- 声明中的 &：表示引用类型(属于类型的一部分)

- 还有按位与 &：a & 1 是位运算


# 流式输入输出

有以下几种流类体系:
- 标准控制台流(`<iostream>`):如`std::cin`,`std:cout`等
-  文件流(`<fstream>`):
    - `std::ifstream`: 文件输入流(读文件)
        
    - `std::ofstream`: 文件输出流(写文件)
    - `std::fstream `文件输入输出流(读写文件)
-  字符串流(`<sstream>`):
    - `std::istringstream`:字符串输入流
    - `std::ostringstream`字符串输出流
    - `std::stringstream`字符串输入输出流
由于比较复杂所以简单来说有以下几点:
-  写文件用 ofstream，操作和 cout 一样
-  读文件用 ifstream，操作和 cin 一样
- 读写都用 fstream
- 所有都可以追加基类`std::ios`,包括但不限于:
   - `std::ios::in` —— 读模式
   - `std::ios::out` —— 写模式
   - `std::ios::app` —— 追加模式(每次写入都在文件末尾)
   - `std::ios::trunc` —— 截断模式(打开时清空文件)
   - `std::ios::binary` —— 二进制模式
   - `std::ios::ate` —— 打开后立即定位到文件末尾


# 内存
1. file scope && local scope(系统自动管理的内存)
Scope中文名为作用域,指的是在源代码中，一个名字(变量、函数等)在哪个区域内是可见、可访问的

其中 Local scope(局部作用域)通常指块作用域：从声明点开始，到最近的` } `结束

比如一个函数中非静态局部变量只能够在函数的局部作用域中存活和调用

值得注意的是存活和可访问不是一个意思,对于一个变量的存活有专门的名称,叫生命周期(Lifetime/存活状态),非静态局部变量的存储期(生命周期)是自动存储期。生命周期不等于作用域,举个例子,存在一种静态局部变量(通过`static int counter = 0`定义),其作用域仍是局部的，生命周期却是整个程序运行期间(静态存储期),值只可以在函数存在的时候访问,但是在两次调用间保持相同,也就是函数结束之后所在内存不会被销毁而是会保留

File scope(文件作用域)是指在函数、类等外部定义的全局变量/函数所具有的特质,作用域从声明点直到文件末尾，生命周期是静态的(程序开始时创建，结束时销毁)

2. 动态内存管理
有一种(第三种)储存期形式为dynamic extent,其内存系由程序的空闲空间 ( free store ) 分配而来，有时也称为 heap memory ( 堆内存)。这种内存必须由程序员自行管理定义方式为:`new Type ( initial_value );`,比如:`int *pia = new int [ 24 ]`(这代表了划出了一个24个整数的数组),`int *pi = new int ( 1024 ) ;`(这代表画出了一个数`pi`,数值为1024)。

我们可以通过delete删除,比如`delete [ ] pia`或`delete pi`。如果最终由 heap 分配而来
的对象也没有被手动释放,这称为 memory leak ( 内存泄漏 )。


# 多文件编译

在现代工程c++中,经常要面对多个c++文件,除了主文件(包含main)我之外,我们会将一些函数等封装到别的文件中,下面我们会介绍流程和特性。

1. 首先c++一般后缀名有`.h`,`.hpp`,`.cpp`,`.o`,前两个是头文件,负责声明,第三个是源文件,负责定义,最后一个是目标文件(中间产物,机器码)。声明不分配内存,只告诉编译器有函数/变量/类,类似目录,负责占位。而定义分配内存,实际创造了了函数/变量。

2. 在首阶段,预处理器会处理所有以`#`开头的部分,然后溯源展开,比如遇到`#include<math.h>`,将会直接把`math.h`复制展开到该处,使用`#define`定义的宏也会在此时被展开,我们就得到一个展开的文本文件(`.i`文件,一般只在内存中处理),这被称作"翻译单元(Translation Unit)"。

3. 接下来会进行编译和汇编(编译器和汇编器),将文本先变为汇编语言(编译器)再变为机器码(汇编器)。其重要的特性是"独立性"也就是编译器一个进程只处理一个文件,进程之间互不干预,如果有跨文件的引用,会引入一个占位符,而不是跨文件去获取函数/变量,每一个`.cpp`文件会变为`.o`文件。

4. 最终通过链接器进行链接,其作用是将不同的`.o`进行缝合,比如遇到占位符就回去声明的文件的位置寻找,最终生成可执行文件。

5. 在走完流程之后,我们会发现两个比较大的漏洞:
   1. 一个出现在第一步,如果有重复引用,比如`A.h `包含了 `B.h`,而 `main.cpp `同时包含了 `A.h` 和 `B.h`,展开的时候`B.h`就会在`main.cpp`中粘贴两次,如果`B.h`中定义了一个结构体,那就很容易出现`error: redefinition of ...`,也就是重复定义,咋办呢?我们可以在所有的头文件加上`#pragma once`,也就是这个头文件在同一个`.cpp`中只能粘贴一次。
   2. 另一个出现在`.hpp`/`.h`中,这两者没有本质区别,唯一的区别是后者是C中的,但是C++通用,我们一般认为,后者使用在纯C或者C/C++混合的项目中,并且只有声明,而前者一般是纯C++项目,很有可能头文件中既有声明也有具体实现。现在因为某些原因,我们不想要别的`.cpp`了,我们想纯头文件库,但是如果两份文件同时包含了头文件,在链接器组装的时候就会出现两次这个函数,崩溃报错`Re-definition error`,这就需要使用`inline`内联。具体就是在`.h`/`.hpp`中在类型前面加上`inline`,用以允许函数定义在多个编译单元中并存，豁免单一定义原则。当然,inline的本义其实是类似宏的展开,也就是建议函数本处展开。





# inline
刚刚讲过了`inline`的现代常见用法,下面来讲一讲其本意。为了讲解这个,我们首先要讲一个函数被调用,背后的原理究竟是什么?首先CPU的执行顺序应该是按照main函数从上到下的,如果遇到函数,那么一般的我们需要为该函数划出一个新内存,首先明确,不同函数之间内存的关系是同栈的,这很好理解,最先开始的函数肯定最后结束。而内存的划分是由两个寄存器负责的,一个`RSP`寄存器,负责监视当前栈的栈顶位置,另一个`RBP`寄存器,负责当前栈底的位置。寄存器是为了解决从内存读写过慢而不满足CPU计算速度的位置在CPU内部的数量极少,但速度极快的极小存储单元。在运行中,每遇到一个函数,就会为其开一个内存,并放置在栈底,称为栈帧,而CPU每次会对栈顶的栈帧做运算,那么每次新开一个栈帧就一定有几个问题:
- 首先`RBP`指针和`RSP`指针(寄存器内部数据类型是指针)需要挪动,重新划定内存的物理边界
- 同时需要在栈顶的栈帧底部加入返回地址方便跳转回去
  
  >注意,大部分现代计算机栈是从高内存地址向低内存地址生长,所以也可以说在"在栈顶塞入一个'返回地址指针'",括号外所谓的栈底指的是相对位置上,也就是在两函数交界处,所以是后进入的栈的"栈底",但是另一方面,从地址和绝对位置,其实应该是栈顶(更加严谨合理),有点像钟乳石的生长和消逝

- 另一个问题是这样会打断CPU的流水线指令,现代计算机的CPU执行指令遵循"工序":
    $$\text{取指令（Fetch）} \rightarrow \text{译码（Decode）} \rightarrow \text{执行（Execute）} \rightarrow \text{写回内存（Writeback）}$$
    为追求速度,讲求的是工业流水线机制,也就是当第一条指令在进行"译码"时,第二条指令就已经进入"取指"阶段了,各个工序同时开工,若要调用普通函数,CPU已经预测执行的内容必须强行刹车，清空当前流水线上所有已经进行到一半的后续指令(称为流水线冲刷,Pipeline Flush),然后去掉头执行新函数的代码
其实严格来说,上述问题并不算大,但是如果函数本身非常简短,而你又大量重复调用函数,这些问题就非常明显:执行时准备工作的时间远超计算本身。为了解决这些问题,其实最好的办法就是不新划内存,这就`inline`的作用,`inline`会使得函数在调用处展开到`main`里面,效率会大大提高,但是这只是一个建议,建议编译器直接展开,一般来说现代编译器非常的智能,他会自己判断需不需要,所以你的建议可能不会被采纳,也就是为什么这个功能现在不是很常用了。

当然,现代CPU其实还有一些一些别的设计,比如“分支预测器”和“返回栈缓冲区（RSB）”,它们会猜你下一步去哪，提前把指令取进来。因此，单纯的“函数跳转”开销在现代CPU上已经被极大地平摊了。因此`inline`还有一个隐藏的好处,就是编译器实际上对于上下文也是会全局优化的(比如常量折叠、死代码消除),所以内联会特别常见。

关于栈还有一点补充,也就是栈溢出(stack overflow),想象你在写深搜,由于其代码特性,每次函数调用是嵌套的,也就是从`dfs(1)`开始,除非搜索结束,否则不会结束每一层的`dfs`,那么从`(1)`开始,一直到`(n)`会全部堆在栈中。如果数据不好,或者忘了停止,递归层数太多就会直接爆栈,也就是你堆叠的内存大小超过了栈本身的大小,被系统直接杀掉。怎么办呢?方法是不使用栈,而是用`堆(heap)`去手动模拟一个栈。栈的特点是:
- 极快：CPU 有专门的硬件指令和寄存器来管理它，分配内存只需要移动一下指针。
- 自动：函数开始时自动申请，函数结束时自动释放，人类不需要操心。
- 缺点：空间很小（通常只有几 MB），且数据生命周期必须和函数绑定。

也就是程序自动管控的内存其实就是栈,而堆的特点是:
- 极大：你电脑内存有多大，它理论上就能用多大。
- 手动：在 C++ 里用 new 申请，用 delete 释放。如果你申请了忘记释放，就会导致著名的内存泄漏（Memory Leak）。
- 缺点：很慢。因为要在浩瀚的仓库里寻找哪里有空闲的空间，还要登记记录。

也就是之前学的申请内存实际上就在这里
    
# 宏(marco)
刚刚一直在讲宏,现在来做一个简单解释。"marco"来自希腊语"makros",指的是"长大",而所有的"宏",本质上都是自动化与文本膨胀。在c++中,所谓宏就是文本替换工具,在预处理器处理时进行的替换。而Office中的宏是一种运行时的自动化VBA脚本,通过自动化帮你处理重复操作,而游戏中的宏则是一种宏案件,用来做按键指令序列映射。

# 重(chong)载函数(Overloaded Function)
这是一种语法糖,方便函数命名,允许在同一个作用域内，存在多个“同名”的函数，只要它们的参数列表（类型、个数、顺序）不同即可。其底层逻辑是"名称修饰(Name Mangling)",编译器在编译时，会把参数的类型缩写强行揉进函数名里。

# 函数模板(Function Template)
我们在多文件编译提到过这个特性,函数模板允许我们在不指定函数变量的类型时写函数,具体语法为:
```c++
// 告诉编译器：T 是一个虚拟的类型代号
template <typename T>
T my_max(T a, T b) {
    return a > b ? a : b;
}

int main() {
    // 隐式调用：编译器看一眼参数，自动推导 T 是 int
    int i = my_max(3, 5); 
    
    // 显式调用：明确告诉编译器，把 T 当作 double 来用
    double d = my_max<double>(3.14, 2.71); 
}

```
注意,模板不能做到声明和定义分离,否则无法被编译为机器码,但同时由于此特性使得我们不需要`inline`声明,而是可以直接在模板里写定义
